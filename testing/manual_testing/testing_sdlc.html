Software Development Life Cycle

Various activities that perform from beginning to the end is called as SDLC. A software development lifecycle is essentially a series of steps, or phases, that provide a model for the development and lifecycle management of an application.

Stages in SDLC:

1. Requirement gathering

2. Analysis

3. Design

4. Implementation or coding

5. Testing

6. Deployment & Maintenance

Requirement gathering:

Business requirements are gathered in this phase. This phase is the main focus of the project managers and stake holders.

The Task of Business Analyst:

Interacts with the customer to gather requirements using the template BRS and submit that template back to the company to make sure that they are on the same page.

Analysis:

A detailed study of the requirements is done in order to check if they are feasible or not.

User Requirements: Describe the tasks the user must be able to accomplish the product. It is captured in use case diagrams and scenario descriptions System Requirements: Detailed software description which can serve as the basis for design or implementation.

Tentative planning: The resource, as well as the time, is temporarily planned here

Technology Selection: The technologies that are required for accomplishing the projects successfully are analyzed and listed off.

Design:

In this phase,the technical details of the design are discussed with the stakeholders and various parameters such as risks, technologies to be used, the capability of the team, project constraints, time and budget are reviewed and then the best design approach is selected for the product.

System Design helps in specifying hardware and system requirements and also helps in defining overall system architecture. The system design specifications serve as input for the next phase of the model.

In this phase, the testers come up with the Test strategy, where they mention what to test, how to test.

Coding:

Developers have to follow the coding guidelines defined by their organization and programming tools like compilers, interpreters, debuggers etc are used to generate the code. Different high-level programming languages such as C, C++, Pascal, Java, and PHP are used for coding. The programming language is chosen with respect to the type of software being developed.

Testing:

After the code is developed it is tested against the requirements to make sure that the product is actually solving the needs addressed and gathered during the requirements phase. During this phase, all types of functional testing like unit testing, integration testing, system testing, acceptance testing are done as well as non-functional testing are also done.

White Box testing: It is a program based testing to estimate the completeness and Correctness of the internal logic of each program structure. It is also known as clear box testing, open box testing, logic driven testing or path driven testing or structural testing and glass box testing.

Black box testing: It is software based testing to estimate the completeness and Correctness of the external functionality of the application. It is also called as Functionality, Behavior, Requirement based or closed box testing.

Deployment and Maintenance:

Once the software has been fully tested and no high priority issues remain in the software, it is time to deploy to production where customers can use the system.

Once a version of the software is released to production, there is usually a maintenance team that looks after any post-production issues.

Some of the SDLC Models are

1. Waterfall Model

2. Prototyping model

3. Incrementing/Overlapping Waterfall Model

4. Spiral Model

5. V-Model

6. Agile model

7. RAD Model

Waterfall Model:

Introduced by Winston Royce in 1970 and is currently the most commonly used model for small software.

Emphasizes that software is developed in sequential phases (plan-> Analyze->Design->Build->Test->Development) with established milestones, documents and reviews at end of each phase.


Traditional waterfall model a one-way linear sequence.

Strengths:

1. Simple to understand and use.

2. Clearly defined stages.

3. Well understood milestones and easy to arrange tasks.

4. Process and results are well documented.

5. Works well for projects where requirements are well understood.

Drawbacks:

1. Requirements must be fully specified at the beginning.

2. Cannot accommodate changing arguments.

3. Not suitable for complex projects.

4. Users can only judge at the end.

Prototyping model:

Usually used in projects where general objectives for software are known, but one cannot identify detailed requirements. Based on IKIWISI(I-Know-It-When-I-see-It) principle.

Strengths:

•  Customer can see the system requirements as they are being gathered.

•  Interaction with the prototype stimulates awareness of additional needed functionality.

•  It serves as a marketing tool.

•  Improves communication between developers & customers.

Drawbacks:

•  Hard to plan the entire project at once leading to difficulties.

•  May encourage excess of change requests.

•  Overall maintainability may be overlooked.

Incremental Model

It is a type of evolutionary model This objective is to develop the system in incremental parts. Requirements are prioritized and then implemented in groups "divide and conquer" Each subsequent release of the system adds function the previous release until all designed functionality has been implemented.


Increments continue until all functionality is implemented.

Advantages:

1. Some working functionality can be developed quickly & early in the life cycle.

2. Results are obtained early and parallel development can be planned.

3. Less costly to change the scope/requirements

4. Delivers an operational product with each release.

Drawbacks:

1. Although the cost of change is lesser it is not very suitable for changing requirements.

2. Requires good planning and design and careful partitioning of the product.

3. Requires early definition of a fully functional product to define the requirements.

Spiral Model

Evolutionary model that combines the interactive nature of prototyping with the controlled and systematic aspects of waterfall life cycle model, along with an aspect of waterfall life cycle model, along with an emphasis on the use of risk management techniques After all the risks have been addressed, the spiral model terminates as a waterfall life cycle.

It is an iteration of four stages.

1. Planning: Identifying the objectives, alternatives, constraints for an iteration i.e; gather requirements for this iteration.

2. Risk Analysis: Identify and analyze risks. Use prototyping, simulations, bench marking and other risk resolution techniques to determine to GO/NO GO for the project.

3. Development and test: If GO, then build the increment of the system.

4. Get Evaluation and feedback/suggestions from sponsors.

Strengths:

1. Flexibility and accommodation to changes.

2. Ability to integrate new technology and architectures

3. Ability to react to risks at each evolutionary level.

4. The Software is produced early in the software lifecycle.

Drawbacks:

1. More complex planning and management process.

2. Cost can be high.

3. Doesn’t work well for small projects.

4. Risk assessment requires highly specific expertise.

V-Model (Verification and Validation model)

A Variant of the waterfall that emphasizes verification and validation of the product. Testing of the product is planned with a corresponding phase of development. Assumes that approximately half of the total development effort will be spent in testing.

Verification:

Verification is testing to confirm that a product complies with its requirements and specifications. It is also called as Unconventional testing.

The standard definition of Verification goes like this: "Are we building the product RIGHT?" i.e. Verification is a process that makes it sure that the software product is developed the right way. The software should conform to its predefined specifications, as the product development goes through different stages, an analysis is done to ensure that all required specifications are met.

During the Verification, the work product (the ready part of the Software being developed and various documentations) is reviewed/examined personally by one or more persons in order to find and point out the defects in it. This process helps in prevention of potential bugs, which may cause failure of the project.

Validation:

Validation is testing to confirm that it satisfies stakeholder needs. It is also called as Conventional testing.

Validation is a process of finding out if the product being built is right? i.e. whatever the software product is being developed, it should do what the user expects it to do. The software product should functionally do what it is supposed to, it should satisfy all the functional requirements set by the user.

Validation is done during or at the end of the development process in order to determine whether the product satisfies specified requirements.

V-Model Strengths

• Emphasizes on planning, verification, and validation in early stages of development

• Rigorous testing efforts

• Easy to use and Easy to track progress

• Works well for projects where requirements are well understood.

V-Model Drawbacks:

• No iteration phases

• Does not easily handle changes in requirements

• Does not contain risk analysis activities

• If any changes happen in midway, then the test documents along with required documents have to be updated

RAD Model (Rapid Application Development)

Developed to respond to the need of delivering systems very fast. It is a methodology for compressing the Analysis, design, build and test phases into as series of short, iterative development cycles Usually used with object-oriented programming such as Java, C++.

It uses time boxing, where some features are pushed to a future release in order to complete the current version Quickly.

It uses time boxing, where some features are pushed to a future release in order to complete the current version Quickly.

Strengths:

• Time to deliver is less.

• Changing requirements can be accommodated.

• Progress can be measured.

• Cycle time can be short with use of powerful RAD tools.

• Use of tools and frameworks.

Drawbacks:

• Management complexity is more.

• Resource requirements are more.

• Suitable for systems that are component based.

• Requires user involvement throughout the life cycle.

Agile Model

Agile: Readiness for motion, quick activity, cleverness in motion.

It is a conceptual framework for software engineering that promotes development iterations throughout the life-cycle of the project. Software developed during one unit of time is referred to as an iteration, which may be lost from one to four weeks. Agile methods also emphasize working software as the primary measure of progress.

Why More IT Companies use Agile model?

Agile approaches, organizations can respond swiftly to customer needs, speed solution delivery and adapt efficiently to change.

Agile Model need:

Improved quality

More opportunities for mid-course corrections

Improved customer or business satisfaction

Better alignment between business and IT

Improved time to market

This methodology follows neither a purely sequential approach nor does it follow a purely iterative approach. It is a selective mix of both of these approaches in addition to quite a few new developmental methods. Fast and incremental development is one of the key principles of this methodology. The focus is on obtaining quick, practical and visible outputs and results, rather than merely following theoretical processes. Continuous customer interaction and participation is an integral part of the entire development process.

Existing Agile methods are

i. Scrum

ii. Extreme programming

iii. Agile unified process

iv. Crystal methodologies & family

v. Features driven development

vi. Adaptive software development

SCRUM: It is a very light framework for delivering real products early often consistently.

Advantages:

• It has an adaptive team which is able to respond to the changing requirements.

• Face to face communications and continues inputs from the customer representatives leaves no space for guess work.

• Documentation is crisp and to the point to save time.

Disadvantages:

• There is a lack of emphasis on the necessary designing and documentation.

• The project can easily get taken off track. If customer representative is not clear. What final outcome that they want.

• Only Senior Programmers are capable of taking the exact kind of decisions, required during the development process.
