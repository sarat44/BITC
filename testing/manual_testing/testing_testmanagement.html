Test Management

Test management ensures that the testing is done effectively and efficiently for software. Testing is a complex activity. Testing is often a distinct sub-project, within the large software development maintenance or integration project.

Test Management includes

1. Test Organisation & Independence.

2. Test planning and Estimation.

3. Test progress monitoring and control.

4. Managing change or configuration management.

5. Risk management.

6. Incident management.

1) Test Organisation & Independence:

The approach towards organizing a test team depends upon the organizational structure. The organization needs to create an environment where testers can deliver an independent, objective assessment of quality.

2) Test Planning and Estimation:

It assesses the software application risks and then develops a plan to determine if the software minimizes those risks. A test plan guides us in the testing activity and helps us confront challenges associated with testing.

Entry and Exit Criteria:

Entry Criteria:

Entry Criteria define when to start testing such as at the beginning of test level or when a set of tests ready for execution.

Entry Criteria Examples:

• Test Environment availability and readiness

• Test tools readiness in the test environment

• Test Code Availability

• Test Data Availability

Exit Criteria: Exit Criteria define when to stop testing such as at the end of the the test level or when a set of test has achieved a specific goal.

Exit Criteria Examples:

• Lots of tests have been run

• Tests have been reviewed/Inspected

• No faults found in current tests

• All faults found fixed and retested

• Specified coverage achieved

• All test run after the last fault fixed, No new faults.

Test Estimation: Estimation & Budgeting involves computation of the resources required for completion of the project. It involves determining what effort and time will be needed to accomplish the testing goals as stated in the test plan. Estimation is complex as every project is unique in nature.

3) Test progress monitoring and control: Test Lead monitors the progress throughout the process against the plan. Any deviations are their test lead will take necessary actions.

Test Lead can directly control

• Number of test iterations

• Test included in iteration

• Entry/Exit criteria applied

•Release Date

Test Lead can control indirectly:

• Rework effort

• Which faults to be fixed (first?)

• Quality of fixes (entry criteria to retest)

Test Lead cannot control

• Number of faults already there

4) Software Configuration Management Software Configuration Management (SCM) is a set of activities designed to control change by identifying the work products that are likely to change, establishing relationships among them, defining mechanisms for managing different versions of these work products.

Why SCM?

It’s very easy for a stream of uncontrolled changes to turn a well run software project into disorder. For that reason, SCM is an essential part of good project management and solid software engineering software practice.

5) Risk Management:

Risk: The chance of injury, damage or a potential hazard.

Risk Analysis: The process of identifying, estimating and evaluating risk.

Risk management: The process of controlling risk and monitoring the effectiveness of the control mechanisms.

There are two types of risks

1. Project risk

2. Product risk

Product risk: These are potential failure areas in the software or system as they are a risk to the quality.

Product risks are as follows

• Error prone software delivered

• The software could cause harm to an individual or company

• Poor software characteristics (eg: Functionality, severity, reliability, usability, and performance)

• Software does not perform its intended functions

Product risk-solution

A risk based approach to testing is a proactive opportunity to reduce the levels of product risk, starting in the initial stages of a product

Project risk: These are related to the projects capability to deliver.

Project risks are as follows:

• Skill and staff shortages

• Personal and training issues

• Political issues (Testers have problems communicating their need and test results)

• Improper attitude toward or expectations of testing (e.g. not appreciating the value of finding defects during testing)

• Supply issues (Failure of a third party, Contractual issues)

Technical issues

• Problems in defining the right requirements

• The extent that requirements can be met given existing requirements

• The quality of design, code and tests

Project risk-solution

When analyzing, managing and mitigating these risks the test manager is following well-established project management principles.

Risk Management

• Risk management activities provide a disciplined approach like

• Assess what can go wrong (risks)

• Determine what risks are important to deal with

• Implement actions to deal with those risks

• Identify new risks (with the help of testing)
