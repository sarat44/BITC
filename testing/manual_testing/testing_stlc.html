Software Testing Life Cycle (STLC)

Below are the phases of the STLC and the activities done during each phases.

The phases of software testing life cycle are:

  • Requirement analysis
  • Test plan
  • Test case development
  • Testbed setup and Dry run
  • Test case execution
  • Defect retesting
  • Test closure

Requirement analysis
This is the first step of STLC. Software quality assurance team or the testing team starts analyzing the required documents to understand the client/end user expectations. During this process, QA team raise questions to the business analyst and other stakeholders if they face any difficulty while analyzing the requirements.

Entry Criteria:
Signed off requirement document should be available.

Exit Criteria:
All the queries raised by the QA team should be answered.

Deliverables:
Generally, there are no deliverables in this phase. But QA team should maintain a query tracker and can show it to the client. Also, they can prepare a report based on approximate automation testing percentage and manual testing percentage.

Test plan
After requirement analysis, QA team starts planning. All the planning should be documented. Generally, cost and effort estimation is done before project starts but sometimes cost and effort estimation is also a part of this phase.

Entry Criteria:
Requirement analysis is completed.

Exit Criteria:
Test Plan document is prepared.

Deliverables:

  1. Test Plan Document
  2. Test Strategy Document
  3. Estimation sheet (Optional)

Difference between Test Plan and Strategy:
In some projects, these two are maintained separately. But in most cases, people include test strategy as a part of test plan document. Typically test strategy is a very high-level approach to testing. On the other hand test plan is a detailed document describing each and every process of testing.

Components of a test strategy document:
If test strategy document is maintained separately the below components are the must. Except these, you can add as many as you want, but keeping it short is recommended.

  1. A process flow diagram of the project
  2. In scope
  3. Out of scope
  4. Test deliverables
  5. Test tools to be used

Component of a Test Plan document:

  • Introduction/Purpose of the document
  • Test items (In Scope)
  • Test deliverables
  • Features not to be tested (Out of Scope)
  • Test Environment requirements (this includes testing tools as well)
  • Test techniques (Manual and automation details)
  • Test phases (Smoke testing, System-Integration Testing, Regression Testing, Sanity Testing etc.)
  • Test Case pass or fail criteria
  • Defect reporting and tracking process
  • Plan of training schedule and details
  • Roles and Responsibilities
  • Testing Schedule (Planned schedule for all testing phases)
  • Risks and Mitigation plan
  • Communication and status reporting process
  • References
  • Approvals
  • Change and configuration management

Test case development
After planning test case development starts. Testers write manual or automation scripts for testing. Test Scenario, Test Cases and Requirement Traceability matrix (RTM) are prepared in this phase.

Entry Criteria:
Test Plan is signed off

Exit Criteria:
Test Scenario, Test Case is ready

Deliverables:

  1. Test Scenario
  2. Test Case
  3. Requirement Traceability matrix (RTM)

Requirement Traceability matrix (RTM):
It is a matrix where each requirement is mapped with test cases. This matrix is prepared to ensure that all the requirements are covered by test cases. The typical process to create this matrix is, in a spreadsheet, all requirement numbers are written in one column and in the next column corresponding test case name/ID are written. This way you can showcase your client, that all the requirements are covered for testing.

Component of a Test Scenario sheet:

  1. Scenario ID
  2. Scenario Name
  3. Scenario Description

Component of a Test Case sheet:

  • Scenario ID
  • Test Case ID
  • Test Case Name
  • Test Case Description
  • Step No
  • Step Description
  • Expected Result
  • Actual Result

Note: For a particular scenario there can be one or multiple test case. So to maintain Scenario ID in Test case sheet is recommended.

Testbed setup and Dry run
Testbed or test environment setup is the next step. Before starting test execution test environment setup is required. In most of the cases, separate environment team does this setup with the information provided by testers. In this phase, there are no deliverables.

Test Execution
In this phase, testers start executing of the test cases they wrote during test design phase. Multiple stages are involved in this execution phase. Let’s describe one by one in order.

Smoke Testing: This is the first phase of the execution. In this phase testers test the environment to check that the environment setup. Generally, testers run their scripts to verify that environment is good to go with further testing. If smoke testing passes, then testers start the actual testing of the new built.

System Integration Testing (SIT): It’s a combination of system testing and integration testing. The main testing happens in this phase. Testers run all their manual and automation test scripts to verify the new system developed.

Regression Testing: This is required when some existing functionality was already there. Testers do regression test to make sure the existing functionality is working properly and it’s not broken due to the new part developed. For a project started from scratch, this testing is not required.

Sanity testing: The last phase of execution. Generally, people do not do this phase. But this is one of the important testing phases to be covered. In this phase, testers identify few critical business area and test that area in depth (Though these are covered during SIT still some more detail testing is required).

Entry Criteria:

  1. Test cases are ready
  2. Testbed setup is done

Exit criteria:
All the test cases are executed

Deliverables:

  1. Test logs
  2. Defect matrix

Defect retesting
This is an extended part of test execution.

Retesting: While SIT and Regression testing testers raise defects. Developers fix them and they redeploy the code in a testing environment. Then tester again tests the failed test cases. This is known as retesting.

Entry criteria:
All the defects are fixed

Exit criteria:
All the failed test cases are executed and they are passed

Deliverables:

  1. Retesting logs
  2. Closed defect matrix

Test Closure

This is the final and end stage of testing life cycle. Generally, tester’s job finishes in this phase and Test lead prepares a detail closure report.

Component of a Test Closure report:

  • Document Objective
  • Introduction
  • Scope of Testing (To show all scopes mentioned in test plan are covered)
  • Testing Schedule (Report planned start date, end date and actual start date and end date of each phase)
  • Scenarios Covered
  • Types of Testing Conducted
  • Test Cycles and Activities
  • Test Environment / Tools Used
  • Test Results and Execution Metrics
  • SIT Status and Defect Metrics
  • Performance Testing Report
  • Test Deliverables

Entry Criteria:

  1. Execution completed.
  2. All the defects are closed.

Exit Criteria:
Test Closure report to be signed off by client.

Deliverables:
Test Closure report.
